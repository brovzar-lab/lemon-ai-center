# LEMON-AI-CENTER — Principal Engineer Audit

**Date:** 2026-07-08
**Scope:** Full codebase, read-only. Quick mode (recon + engineering analysis; no browser).
**Method:** 4 parallel code sweeps (routes, server internals, frontend, deps/config) + hand-verification of every finding that made the list.

---

## Phase 1 — Recon

**What it is.** A single-user CEO command center: one dashboard over Gmail + Google Calendar + an Obsidian "Brain" + Anthropic AI, plus a DEVELOPMENT-HELL "slate" module for screenplay tracking. Vite + React 18 + Zustand + Tailwind frontend; Express (TS→CJS) backend as an auth/proxy layer; Firebase/Firestore for data and sessions; Google OAuth. One Railway service behind a Cloudflare Tunnel at ceo.billyrovzar.com.

**Core flow.** Google OAuth (server-side, gated by `ALLOWED_EMAILS`, `auth.ts:109`) → session cookie `sid` in a Firestore store → server mints a Firebase custom token (`index.ts:195`) → client `signInWithCustomToken` (`firebaseAuth.ts:14`) so Firestore reads pass owner-scoped rules. Views hydrate from REST (`/api/*`) and live Firestore `onSnapshot` subscriptions. Scheduled jobs run via external Railway Cron → `POST /api/engine/cron/:jobId`.

**State & secrets.** State: Zustand (client) + Firestore (server of record). Secrets: encrypted Google refresh tokens (AES-256-GCM, `encryption.ts`) in `google_tokens` (server-only); `TOKEN_ENCRYPTION_KEY` never in Firestore; session secret enforced in prod. No secrets committed (`.env` gitignored; `.env.example` clean).

**Ambition.** A trustworthy at-a-glance briefing tool for a CEO with ADHD who relies on it to surface what matters and never mislead. Judged against that bar, the biggest risks are the ones where the app shows *confident wrong data*, not perf.

**Confirmed healthy (recon dispelled these):** no seed/demo data shown as real (ripped out); a single Firebase app, not two; no duplicate deps (no axios, one icon/state/search lib); strict TypeScript on both sides; encryption is textbook; no unauthenticated `/api/*` data routes; no committed build artifacts or secrets.

## Phase 3 — Top 10 (ranked by impact × urgency ÷ effort)

### 1. FIX — Firestore rules authorize by authentication only; the confidential corpus is readable by any Firebase-authed identity
- **What:** `isAuthed()` checks *only* that a user is signed in, so the ingested Drive corpus, Obsidian notes, and document metadata are readable by anyone who can obtain any auth identity on this Firebase project.
- **Evidence:** `firestore.rules:5-7` (`isAuthed` = `request.auth != null`); reads gated by it at `:11` (`allowed_users`), `:65-66` (`system/corpus/chunks`), `:69-70` (`system/corpus/files`), `:74-77` (`system/obsidian/notes`), `:81-83` (`system/{docId}`), `:87-89` (`documents`). App only ever authenticates via server-minted custom token (`firebaseAuth.ts:14`, `auth.ts:109`), but the public `VITE_FIREBASE_*` config ships in the bundle, so any *other* enabled sign-in provider (Google, email/password, anonymous) bypasses the server entirely.
- **Why it matters:** The corpus includes the confidential CFO "carpetas" (167 financial dossiers) and the Brain (people, legal, deals). If anonymous or open sign-up is enabled on the project, this is an internet-readable breach of the most sensitive data in the company — reachable directly from the Firestore client SDK, skipping every server check.
- **The fix:** (a) **Today, 2 min:** Firebase console → Authentication → Sign-in method — confirm only the providers you control are enabled and self-serve sign-up is off. (b) **Code:** add the user's email as a custom claim when minting the token (`createCustomToken(uid, { email })`, `index.ts:198`), then gate every `system/*` and `documents/*` read on allowlist membership — e.g. `function isAllowed() { return isAuthed() && exists(/databases/$(database)/documents/allowed_users/$(request.auth.token.email)); }` in `firestore.rules`. Alternatively serve that corpus only through the already-authenticated Express API and set those reads to `if false`.
- **Effort:** S
- **Risk if ignored:** ticking bomb (conditional on Auth config — but the rule should never depend on that config staying right)

### 2. FIX — Delegation emails are open to MIME header injection (the twin route was already hardened; this one was missed)
- **What:** `to`, `toName`, and `taskTitle` are interpolated raw into email headers with only a presence check, so a `\r\n` payload can inject headers (e.g. a blind `Bcc:`).
- **Evidence:** `delegations.ts:36-37` builds `` `To: ${toName} <${to}>\r\nSubject: Action needed: ${taskTitle}\r\n...` `` after only `if (!to || !taskTitle)` at `:24`. The sibling send route already fixed exactly this: `gmail.ts:13` (`sanitizeHeader` strips CRLF) + `gmail.ts:18-23` (Zod validates `to` as an email). The `catch {}` at `delegations.ts:60` then swallows any resulting error with no log.
- **Why it matters:** Task titles and contact names in this app can originate from AI-extracted or inbound-email content, so a crafted value could silently blind-copy every delegation out of your authenticated Gmail — and you'd never see an error.
- **The fix:** Import `sanitizeHeader` from `gmail.ts` (or lift it to a shared util), add a Zod schema mirroring `SendSchema` (validate `to` as email; strip CRLF from `toName`/`taskTitle`), and replace the bare `catch {}` at `:60` with a logged error.
- **Effort:** S (~20 min) — **quick win**
- **Risk if ignored:** high

### 3. FIX — Failures render as confident truth ("Everything looks good", fabricated zeros, empty listeners)
- **What:** Multiple data paths swallow errors and present success/empty as real, across both REST stores and Firestore subscriptions.
- **Evidence:** `useInboxStore.ts:30-32` catches and leaves `threads: []` with no error flag → `InboxIntelView.tsx:232` prints "Everything looks good — nothing urgent right now"; same silent pattern in `useCalendarStore.ts:21-23` and `useSparkStore.ts:22-24`. `today.ts:89-91` wraps two full scans in `catch {}` and returns `{done:0,queued:0,...}` as HTTP 200. 7 of 9 Firestore-subscribing stores pass no `onSnapshot` error callback (`useMissionStore.ts:73-114` 0/7, `useTrackersStore.ts:90-109` 0/6, `useTaskStore.ts:44`, `useCaptureStore.ts:20`, `useActionLogStore.ts:22`, `useDecisionStore.ts:33`).
- **Why it matters:** For a tool whose entire value is trust-at-a-glance, a silent Gmail/Firestore failure telling you "all clear" is worse than a crash — you act on a lie and miss the urgent thing.
- **The fix:** Give each store an `error` field; set it in every catch and in `onSnapshot`'s error callback. Render an error state (not empty state) when `error` is set — especially in `InboxIntelView`. Remove the fabricating `catch {}` in `today.ts` and return the `{ error }` envelope.
- **Effort:** M
- **Risk if ignored:** high

### 4. FIX — A dead Google token shows "unavailable, try again" forever, never "reconnect Google," and is never cleared
- **What:** When the refresh token dies (`invalid_grant`), the error maps to a generic retryable 500 instead of a reconnect prompt, and the stale token is left in place.
- **Evidence:** `googleAuth.ts:24` throws on refresh failure; `tokenCache.ts:42-45` rethrows without clearing; `clearAccessToken` (`tokenCache.ts:22`) is never called in production. Route handlers only special-case 403 (`gmail.ts:78-82`, `calendar.ts:49-53`), so `invalid_grant` (HTTP 400) falls through to `UPSTREAM_ERROR, retryable:true`. In background jobs (`inboxScan.ts:96`, `nightly.ts:82`) it throws to the ledger silently.
- **Why it matters:** A lapsed Google consent turns the whole dashboard into permanently stale/empty data with a misleading "try again later," and the only escape (re-auth) is never suggested. Pairs with #3 to make the failure invisible.
- **The fix:** Detect `invalid_grant` in `getOrRefreshToken`, call `clearAccessToken` + delete the stored refresh token, and return a distinct `{ code: 'REAUTH_REQUIRED' }`. Client shows a "Reconnect Google" banner linking `/auth/google/start`.
- **Effort:** S–M
- **Risk if ignored:** high

### 5. FIX — Boot freezes the event loop (git clone + synchronous vault scan) → `/health` hangs → deploy crash-loop risk
- **What:** Two blocking operations run inside the `app.listen` callback, so Node can't answer `/health` during boot.
- **Evidence:** `index.ts:242` calls `initVaultSync()` which runs `execSync('git clone --depth 1', {timeout:120_000})` (`vaultSync.ts:44`); then `index.ts:245` awaits `initBrainEngine`, whose `scanVault()` walks the tree with synchronous `fs.readFileSync` + `gray-matter` parse per file (`brain/scanner.ts:74-124`). Both block the single thread; `/health` (`index.ts:50`) can't run meanwhile. Railway healthcheck timeout is 120s (`railway.json`).
- **Why it matters:** As the vault grows, boot time approaches the healthcheck timeout; Railway then marks the deploy unhealthy and restarts → a crash-loop where the app never comes up. Already a latent risk at 4,400 docs.
- **The fix:** Move vault sync + brain init out of the listen path — respond to `/health` immediately, run clone/scan after `listen` via `setImmediate`/async, and gate `/api/ready` on brain readiness. Replace `execSync` with async `exec`/`spawn`; make `scanVault` async (`fs.promises`).
- **Effort:** M
- **Risk if ignored:** high

### 6. ADD — No alerting on any job failure; morning assembly can fail silently at 5:30am
- **What:** Job failures write a Firestore ledger row and a `console.error` and stop — no retry, no notification channel anywhere.
- **Evidence:** `engine/index.ts:131-143` (ledger `status:'error'` + `console.error`, no retry/backoff/alert); `morningAssembly.ts:42-44` aggregates and rethrows. Grep across `server/**` for slack/twilio/sendgrid/webhook/pagerduty → no notification integration. Failures surface only if you open the dashboard and a banner renders (and per #3/#8 that banner can itself be suppressed).
- **Why it matters:** The morning briefing is the product's headline moment; if it dies (Anthropic outage, dead Google token) you wake to an empty Today with zero signal that anything broke.
- **The fix:** In `runJob`'s catch, send one notification (email via the existing Gmail client, or a webhook) on job failure. Optionally one retry with backoff for transient API errors. Record `lastRunAt`/`lastError` and surface a persistent banner that cannot be null-rendered away.
- **Effort:** M
- **Risk if ignored:** high

### 7. UPGRADE — AI runs a generation behind, with 12 model IDs hardcoded across 10 files while a correct registry sits unused
- **What:** Most AI calls pin `claude-sonnet-4-6` and one pins `claude-opus-4-7` — both a generation behind current (Sonnet 5, Opus 4.8) — and the IDs are scattered instead of centralized.
- **Evidence:** `aiChat.ts:10`, `draftReply.ts:11`, `voice.ts:12`, `priorityEngine.ts:72`, `brief.ts:25-26` (`claude-sonnet-4-6`); `brief.ts:24` (`claude-opus-4-7`); plus `haiku-4-5-*` in `spark.ts:11`, `tasks.ts:12`, `corrections.ts:47`, `precompute.ts:35`, `relationshipContext.ts:48`. Meanwhile `shared/models.ts:5-10` already defines a registry (`smart: claude-opus-4-8`, `fast: claude-haiku-4-5`) that only the slate module uses; its own comment admits "the rest of the app still hardcodes previous-generation ids per file."
- **Why it matters:** The core CEO-facing outputs (briefings, chat, priority ranking) run on older models, and a model refresh today means editing 10 files instead of one line.
- **The fix:** Point `CLAUDE_MODELS` at current IDs (`smart: claude-opus-4-8`, add a `balanced: claude-sonnet-5` tier, `fast: claude-haiku-4-5-20251001`), then replace every hardcoded string with a registry import. Confirm exact IDs against Anthropic's current model list at change time.
- **Effort:** S–M
- **Risk if ignored:** medium

### 9. FIX — CSRF layer: PUT bypasses the check, the minted token is never validated, and a public-wildcard origin is trusted
- **What:** The origin check skips `PUT`; the `/api/csrf` token endpoint is dead code; and `*.trycloudflare.com` (a domain anyone can mint) is in the allowlist.
- **Evidence:** `csrfCheck.ts:3` `WRITE_METHODS = {POST, PATCH, DELETE}` — no `PUT`, so `PUT /api/voice-profile` (`voice.ts:55`) runs the middleware as a no-op. `index.ts:181-186` mints a CSRF token that no server code ever reads (grep: written, never compared) while clients dutifully round-trip it (`useTodayStore.ts:93`, `ReplyModal.tsx:49`). `csrfCheck.ts:8` and `index.ts:137` both trust `/\.trycloudflare\.com$/`.
- **Why it matters:** CSRF defense currently rests entirely on the `sameSite:'lax'` cookie; the middleware gives false assurance, and the dead token adds latency and complexity for zero protection.
- **The fix:** Add `'PUT'` to `WRITE_METHODS`; delete the `/api/csrf` endpoint and its client round-trips (or actually validate the token if you want defense beyond same-site); drop the `trycloudflare.com` wildcard from both allowlists now that prod is on a stable domain.
- **Effort:** S (~10–15 min) — **quick win**
- **Risk if ignored:** medium

### 10. FIX — Brain index uses lossy char-sum hash IDs → wrong-note citations in Claude context, with no staleness visibility
- **What:** Vault paths are mapped to IDs by a 32-bit character-sum hash, and reverse lookup returns the first hash match via linear scan — collisions return the wrong document.
- **Evidence:** `brain/indexer.ts:117-124` (`pathToId` char-sum), `:126-132` (`idToDoc` O(n) first-match); `addDoc`/`removeDoc` act on that id (`:35-52`). `/api/ready` (`ready.ts:32-44`) exposes `docCount`/`chunkCount` but no `lastIndexedAt`, so a stale or wedged index is invisible.
- **Why it matters:** A char-sum hash collides easily (anagrams, reordered paths); a collision injects the *wrong note* as a citation into a Claude briefing — the exact "confident wrong fact" failure your global rules call out (the Pavón error). Silent, and undetectable from ops.
- **The fix:** Replace `pathToId` with a real hash (`crypto.createHash('sha1').update(path)`) or just key `docs` by path directly and store `id→path` in a `Map`; record `lastIndexedAt` in `getStats()` and surface it in `/api/ready`.
- **Effort:** M
- **Risk if ignored:** medium

## Big bet (the one trajectory-changer)
**Make the app constitutionally incapable of showing fake success** — the reliability/trust overhaul spanning #3, #4, and #6. Today the dashboard can tell you "all clear" while Gmail is down, sit on stale data when your Google token dies, and let the 5:30 briefing fail without a peep. Fixing these together converts the product from "looks fine, might be lying" into "trustworthy at a glance" — which is the entire reason this tool exists. It's L-effort in aggregate (error model across ~15 stores + a reconnect flow + one alerting channel), but it changes what the app *is*.

## Honest opinion
**Healthy, and clearly maturing — not fighting itself.** The bones are good: real auth, encrypted tokens, owner-scoped data, a consistent response envelope, 375 passing tests, strict TypeScript, and a genuinely thoughtful architecture (store-per-domain, shared types, a Brain/slate retrieval layer). The scars are the honest kind — commit messages and code comments show real bugs already hunted down (the "empty panels era," the CRLF fix in `gmail.ts`, the session-cookie proxy saga). The weaknesses cluster in one place and it's the same place every time: **the app is too polite about failure.** It swallows errors, fakes success, and never raises its hand — which for a decision-support tool is the one flaw that erodes the whole point. That's a fixable posture, not a rotten foundation. Fix the silent-failure habit and tighten the two security gaps, and this is a genuinely strong internal product.
